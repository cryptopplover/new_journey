项目介绍

Ceramic 建立在 IPFS 之上，是一个去中心化的开源平台，是用于创建、存储、托管、共享和修改数据的分布式数据网络，可以定义为区块链网络和应用之间的数据中间层（或者说是Layer2）。借助 Ceramic 的分布式流（stream）网络，开发人员可以直接在去中心化网络上存储信息流和更新、编辑文件，可用于存储各种动态、可变的内容。WEB3应用可以在ceramic上存储、更新身份数据（个人资料、声誉、社交图）、用户生成的内容（帖子、交互）、动态应用程序数据。
同时，开发者可以在没有数据库服务器的情况下在ceramic上运行web3社交或信息流dapp。
在某种意义上来讲，受限于单个区块的容量，目前的较出名的公链网络本身都不具备高频率、大量的信息流数据储存功能。所以，今天我们看到的公链上跑的都是DEFI应用，区块存储的都是交易数据。
ceramic的解决方案是在IPFS网络存储web3应用的用户数据和信息流数据，DID身份为数据的主人，让数据跟着身份走，被DIDI控制。其次，可以用任何公链为存储在IPFS上的数据打上时间戳，将其存入项目方或者用户选择的区块链网络中达到数据的链上确权。
ceramic的业务实现逻辑：
在 Ceramic 中，所有内容都存储在智能文档stream信息流当中，每个stream流都是都是对应在 IPFS中的日志，每次数据的存储都由去中心身份 (DID) 签名以进行验证，然后锚定在区块链中以达成共识/也可以不上链。因为 Ceramic 建立在P2P网络上，所以所有文档都是公开可发现的，并且可以被其他文档引用或被网络上的任何其他参与者查询。

ceramic三大业务模块

Ceramic认为围绕以用户为中心的数据模型构建Web3应用程序 数据架构非常重要，Ceramic提出的构建主权（WEB3）应用程序的三大基本原理：
  • 具有快速可变性的分散式数据可用性网络
  • 用于以身份为中心、基于模型的数据存储和检索协议
  • 用于在运行时对用户存储执行 CRUD 操作的客户端

1.分布式数据网络

存储：Ceramic 提供了类似于数据库的高级功能，其底层是P2P和IPFS网络，及时不使用区块链的情况下， 也可以达到分布式存储。主要是以stream的形式，采用DAG模式在IPFS网络中存储和托管用户数据或应用程序数据，以消除集中式数据库服务器、区块链或本地存储。stream是存储用户生成的内容、用户活动和历史、应用程序数据和状态等的后端替代品。

ceramic基于流（stream）和DAG的信息存储架构专为 Web2 规模的容量和延迟而设计，可处理任何类型的数据。建立在包括 IPFS、libp2p 和 DID 在内的开放标准之上，并与 Filecoin 或 Arweave 等任何原始存储协议兼容，存储在 Ceramic 上的所有信息都存在于无需许可的跨链网络中，让开发人员可以利用不断增长的身份库和数据，同时使用他们的首选堆栈。

数据的可编辑性

在 IPFS 上存储可编辑、受版本控制的文档、文件和媒体，保持透明且可审计的历史记录，而无需跟踪更改的哈希值。stream 已被用于存储社交媒体帖子、新闻文章、博客和出版物、JSON 文档、模式、元数据、NFT 资产和可验证的凭证。
在 Ceramic 上，每条信息都表示为仅附加的提交日志，称为Stream。每个流都是存储在 IPLD 中的 DAG，具有称为StreamID的不可变名称和称为StreamState的可验证状态。流在概念上类似于 Git 树，每个流都可以被视为自己的区块链、分类帐或事件日志。


目前存储在ipfs以及Filcoin、Arweave上的数据是不可编辑和更新的，Ceramic作为中间层主要解决这样的痛点。适合未来的大规模的Web3社交项目以及信息流项目采用。例如Web3的推特和Web3的Whatsapp、Web3的TicTok等。传统的Layer1公链不合适大量的信息流应用（社交应用）的运行，Ceramic采用stream流和DAG技术进行数据存储和更新。

2.去中心化跨链身份聚合器中间件-IDX

IDX（身份索引）是使用Ceramic 分布式数据库构建的，用于存储的身份数据流以及用于将流链接到用户的跨链身份聚合器协议。它提供了一个去中心化索引（图谱），允许结构化数据与去中心化身份 (DID) 相关联。

IDX提供了一个替代中心化服务器的孤岛身份数据的链上解决方案。其具有有下功能：
  • 通过简化数据的结构并将数据与用户的个人索引相关联，IDX 允许应用程序保存、发现和路由到用户的数据。
  • 支持对所有公链和所有项链项目的身份数据、社交网络图谱、社交动态等DID数据的聚合、存储和读取（通过API的模式）。
  • 所有DID数据都由用户自己掌控，所有数据也都可以被调用和读取。
IDX在ceramic中的作用
  • 它实现了在 Ceramic 之上实现的以身份为中心、基于模型的数据存储和检索的协议。
  • 目前部署在 Ceramic 主网上的几乎所有应用程序都在使用 IDX框架，文档的每次更新都必须由其所有者的 DID 签名，从而为其内容提供可验证性。
  • 通过IDX的钱包签名实现跨应用程序数据互操作性。
IDX是使用Ceramic 建块并启用的第一个产品，由Ceramic自主开发。IDX目前被多个Web3社交、身份项目与ceramic同时采用。

3 3ID connect

IDX是ceramic数据网络连接用户数据流stream和用户身份（DID）的组件和协议。其最终的实现原理为3ID Connect。

  • 3ID DID 方法：3ID 是一种DID方法，可用于对 Ceramic 进行身份验证并对依赖 DID 的流执行写入操作，3ID DID 是一种强大的 DID 方法，支持多个密钥、密钥轮换和撤销。3ID DID 方法在 W3C 的官方 DID 方法注册表中，完全符合去中心化身份标准。
  • 当 3ID 与IDX和3ID 钥匙串（如在3ID Connect中实现）一起使用时，可以使用来自任何 L1 或 L2 网络的任意数量的区块链帐户轻松控制 3ID。这提供了一种在所有其他平台上统一用户身份的方法。
通过3ID Connect。可以使所有区块链钱包地址聚合到一个DID中，用户通过私钥签名，与用户相关的所有数据都可以存进基于ceramic的IDX当中，在所有DAPP中授权使用。做到以用户为中心的去中心化数据库。

7.同步和分享:使用标准 API 查询、同步或订阅任何流
8.将stream的内容备份到 IPFS、Filecoin 或 Arweave

Ceramic的特点与优势

基于IPFS的数据存储
区别于AR或者Filcoin，P2P网络是存储成本更低的方案。每个用户和DAPP项目方都可以成为节点。类似于电驴一样，基于P2P网络的抖音或者微信，在现在已经实现。
数据可编辑性
目前普通文件的去中心化储存已经比较完善。像基于Arweave 的https://mirror.xyz 可以储存文章，或者NFT背后的JPEG文档，也大多存在Arweave或者Filecoin里面。这些资料有一个特点，就是不需要更新，或者不需要频繁地更新。我们称这些数据为immutable data。
因为mutable data的高频更新，目前去中心化存储无法支持。Ceramic提出了一个流（stream）的概念，每个流储存着一个数据库的信息，可以随时更新迭代。一个节点（node）可以完成一个流的验证与计算，而不需要取得所有节点的共识。减轻了整条链的负担，让数据库的去中心化存储成为可能。
数据的可组合性
在Web3的世界里，数据不再孤立而且结构统一。在Uniswap上交易和在Sushiswap上交易不需要经过API转化就可以在Etherscan或者Zapper和Zerion上看到历史交易数据。Ceramic可以引入更多适用于不同场景的数据结构，让基于Web3的大数据分析更加便捷。这点随着未来Web3应用的爆发，可以Ceramic可以发挥出更大的作用。
围绕以DID为中心的用户数据可用性
IDX+3ID，可以让WEB3应用中的数据跟着用户走。有利于用户数据的跨平台，IDX是Web3的去中心化身份的底层设施。Ceramic 的身份系统 IDX 支持用于跨链 ID 的 DID、用于存储的流以及用于将流链接到用户的聚合器。满足于 W3C 标准且支持60多项DID方法，可用于来自许多不同区块链的不同钱包，构建可互操作的、用户控制的数据存储应用程序。

高拓展性
公开的数据源空间允许使用者引用多个流将其聚合成复杂组合，或将现有流分叉形成新的流。这给了应用程序无限的处理空间。

技术原理

ceramic以stream的形式来存储数据。一个stream就是一个存储账簿。
目前支持的stream类型：
  • tile document：一个key-value的文件存储，将用户的k-v数据以json格式存储。
  • caip10-link：一个存储DID与链上地址（？）的对应关系。
  • 用户自定义：还没看到具体实现。（还不确定）
stream是一条数据链，每个节点是一个commit。
有三种commit：
  • Genesis commit：创建stream的提交，stream的根节点，也确定了stream的ID。
  • Raw commit：用户修改数据的提交。
  • Anchor commit：伴随上面的commit发送，发送以太坊上确定此次commit的交易的信息(交易Hash，块Hash，块时间戳)。
ceramic将每个stream的commit以链的形式存储到IPFS上。
综述：ceramic提供了一个去中心化的key-value云数据库服务。

疑问点

  1. anchor commit对效率的影响。
默认提交commit参数
const DEFAULT_UPDATE_OPTS = { anchor: true, publish: true, throwOnInvalidCommit: true }
意味着每一次raw commit都需要进行一次 anchor 提交，需要与以太坊交互。这样以太坊的tps和GAS都将影响ceramic。
2. DAG 存储
从目前的两种stream实现来看，暂时没有所谓的DAG数据结构的存在。
3. 实现方案
tile document 类型的stream是ceramic现在主要的业务应用。但是其实现比较简单，只是用JSON格式维护了一个文本，来支撑key-value的增删改查。效率上相对于专业的nosql数据库，应该是比较低下的。
